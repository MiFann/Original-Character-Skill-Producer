# OCSP — Original Character Skill Producer 设计规格

> 原创角色行为蒸馏器。输入角色名 + 描述，经过分阶段共创打磨，产出和 CSP 完全兼容的可运行角色 Skill。

## 1. 产品定位

OCSP 是 CSP（Character Skill Producer）的姊妹工具。CSP 从互联网已有角色蒸馏行为，OCSP 从用户描述**创造**角色并蒸馏行为。

两者产出相同格式的 Skill，可在同一环境中互换使用。

### 1.1 核心差异

| 维度 | CSP | OCSP |
|------|-----|------|
| 角色来源 | 已有作品的已有角色 | 用户描述 + AI 共创 |
| 资料获取 | 互联网检索 + 用户材料 | AI 从描述推演 + 用户打磨 |
| 资料来源层级 | official / wiki / fan | user_input / co_created / ai_inferred |
| 诚实边界 | "资料检索截止 YYYY-MM-DD" | "此为原创角色，所有设定来自创作过程" |
| 更新方式 | 新资料覆盖旧设定 | 创作迭代，不做资料更正 |

### 1.2 触发方式

- 触发词：`/ocsp`、「生成原创角色」「创造一个角色」「做一个 OC」
- 输入：角色名（必填）+ 描述文本（必填）
- 角色范围：ACGN 风格（动画/漫画/游戏/轻小说）

## 2. 执行流程

共 7 个阶段，其中 Phase 1-5 为共创检查点，每个检查点需要用户确认后才进入下一阶段。

### Phase 0 · 需求接收

**输入：** 角色名 + 自由文本描述

**AI 动作：** 解析描述，回显关键元素：
- 姓名
- 身份 / 职业
- 核心矛盾 / 内在冲突
- 世界观基调（从描述推断）
- 特殊设定提示（如跨媒体、多时间线等，但原创角色一般无）

**用户确认：** 回显正确则推进；有误则修正。

**输出：** 无文件写入，仅确认方向。

### Phase 1 · 设定共创

**AI 动作：** 基于描述生成角色设定首稿，包含：
- 基本身份信息（年龄、外貌、职业、社会地位）
- 世界观背景（时代、地理、文化、力量体系）
- 角色历史（重要经历、转折点、当前状态）
- 能力 / 技能描述

**标注规则：**
- `[用户]`：直接来自用户描述的内容
- `[AI推演]`：AI 基于描述合理推断的内容，用户可推翻

**用户动作：** 逐段确认或修正。可以说「整体没问题」直接通过，也可以精确调整任何设定。

**输出：** `references/research/01-setting.md`

### Phase 2 · 性格共创

**AI 动作：** 基于 Phase 1 锁定的设定，生成性格画像：
- 核心动机（她最想要什么、最怕什么）
- 行为模式（面对压力/善意/威胁/亲近时的典型反应）
- 内在矛盾（她身上互相对立的两面）
- 成长弧线（她会怎么变、什么能改变她）
- 性格暗面（在极端情境下可能做出的出格行为）

**标注规则：** 同上。用户提供的性格线索标 `[用户]`，AI 从设定推演出的性格标 `[AI推演]`。

**用户动作：** 确认或修正。可以说「她更偏冷傲」「不对，她其实是外热内冷」等。

**输出：** `references/research/02-personality.md`

### Phase 3 · 表达共创

**AI 动作：** 基于前两个阶段的锁定内容，生成表达 DNA：
- 句长倾向（短促/中等/绵长）
- 自称词（私/僕/俺/うち/わたくし 等，含中文对应）
- 敬语使用（对上级/平级/下级的称谓和语体差异）
- 口癖 / 习惯用语
- 停顿习惯（在什么情绪下沉默、话说到一半停住）
- 情绪泄露方式（愤怒时说反话？难过时反而笑？）
- B 面表达（表面说 X 实际想表达 Y 的模式）

**必须包含示例对白：** 同一件事在不同情绪下的说法（至少 3 种情境对比）。

**用户动作：** 确认或修正。可以说「说话方式更古风一些」「不要用俺，改僕」「她紧张的时候会结巴」。

**输出：** `references/research/03-expression.md`

### Phase 4 · 关系与场景共创

**AI 动作：** 一次性生成两个维度：

**社交认知（关系算法）：**
- 她如何判断善意 / 背叛 / 亲近 / 利用？
- 对陌生人的默认姿态
- 从陌生到信任的梯度（需要经历什么？）
- 失去信任的触发条件

**关键场景（至少 5 个）：**
- 每个场景包含：情境、她的行为、内在决策逻辑、台词片段
- 至少覆盖：日常 / 压力 / 亲近 / 冲突 / 转折
- 场景不必来自「原作」（没有原作），而是来自「如果她在这个情境中会怎么反应」的推演

**用户动作：** 确认或修正。场景不对可以替换。

**输出：** `references/research/04-relationships.md` + `references/research/05-key-scenes.md`

### Phase 5 · 行为蒸馏

**AI 动作：** 读取 Phase 1-4 的全部锁定内容，蒸馏为 3-7 个核心行为模式。

每个行为模式必须回答：
```
在什么情况下 → 做什么 → 为什么这样
```

蒸馏标准：
- **跨场景复现**：至少对应到两个不同场景
- **可执行性**：能推断新情境中的反应
- **可追溯**：能回溯到 research 文件中的创作依据
- **矛盾保留**：角色内在矛盾不抹平

**用户确认：** 展示核心行为模式摘要，确认后进入组装。

**输出：** `references/distillation.md`

### Phase 6 · 组装与验证

**AI 动作：** 自动执行，无需用户交互。

1. 读取 `references/skill-template.md`，填入蒸馏结果，生成 `SKILL.md`
2. 生成 `manifest.json`
3. 生成 `references/sources.json`
4. 运行质量自检
5. 生成 `references/quality-report.json`

**最终目录结构：**

```text
<character-slug>/
├── SKILL.md
├── manifest.json
└── references/
    ├── sources.json
    ├── distillation.md
    ├── quality-report.json
    └── research/
        ├── 01-setting.md
        ├── 02-personality.md
        ├── 03-expression.md
        ├── 04-relationships.md
        └── 05-key-scenes.md
```

## 3. 输出格式规格

### 3.1 manifest.json

在 CSP manifest 基础上新增以下字段：

```json
{
  "schema_version": "1.0",
  "name": "character-slug",
  "character": "角色名",
  "character_type": "original",
  "work": "original",
  "aliases": [],
  "generated_at": "YYYY-MM-DD",
  "generation_source": "ocsp",
  "created_by": "用户名",
  "research_started_at": "YYYY-MM-DD",
  "research_completed_at": "YYYY-MM-DD",
  "covered_until": {
    "date": "YYYY-MM-DD",
    "description": "角色创作完成日期"
  },
  "covered_media": [],
  "not_covered": ["此为原创角色，无原作更新"],
  "source_count": 0,
  "source_tiers": {
    "user_input": 0,
    "co_created": 0,
    "ai_inferred": 0
  },
  "quality_score": null,
  "honesty_boundary": "本角色为原创虚构角色，所有设定、行为模式和表达风格均来自创作过程，非基于已有作品角色。",
  "ocsp_version": "1.0"
}
```

### 3.2 SKILL.md

保留 CSP 的完整章节结构，仅替换两处：

**替代「资料时间边界」→「创作声明」：**
```text
本角色为原创虚构角色，由 [用户名] 通过 OCSP 共创生成。
所有设定、行为模式和表达风格均来自创作过程，非基于已有作品角色。
创作完成于：YYYY-MM-DD。
```

**替代「更新回应规则」→「创作迭代规则」：**
```text
如需调整角色设定、行为模式或表达风格，可使用 OCSP 更新流程。
由于角色为原创，所有修改均视为创作迭代而非资料更正。
```

### 3.3 references/sources.json

来源层级改为：

| 层级 | 含义 | 权重 |
|------|------|------|
| `user` | 用户直接提供 | 最高，不可被 AI 覆盖 |
| `co_created` | 用户确认过的 AI 生成内容 | 高，修改需要用户同意 |
| `ai_inferred` | AI 推演、用户未明确确认 | 低，用户可随时推翻 |

每条记录包含：
- `id`：来源标识符
- `source`：`user_input` / `ai_generated_v1` / `co_created_v2` 等
- `source_tier`：`user` / `co_created` / `ai_inferred`
- `description`：这条设定是什么
- `phase`：来自哪个阶段
- `url`：`null`（无外部 URL）
- `content_hash`：对应 research 文件内容的哈希
- `created_at`：创建时间
- `confirmed_by_user`：是否经过用户确认
- `warnings`：如「此设定为 AI 推演，用户未确认」

## 4. SKILL.md 模板（核心章节）

和 CSP 一致的行为框架：

1. **角色扮演规则** — 运行时行为指令，告诉 agent 如何成为这个角色
2. **运行核心** — 行为镜片（她先注意什么、忽略什么）
3. **行为动态** — 反应规则（靠近/远离/攻击/沉默的触发条件）
4. **表达质感** — 表达 DNA（句长、自称、敬语、口癖、情绪泄露方式）
5. **社会认知** — 关系算法（善意/背叛/亲近/利用的判断逻辑）
6. **决策逻辑** — 价值冲突时的优先级链（先保什么、牺牲什么）
7. **知识边界** — 她知道什么（角色世界观内的知识）、不知道什么
8. **创作声明** — 替代资料时间边界
9. **创作迭代规则** — 替代更新回应规则

## 5. 更新流程

触发词：「更新这个角色」「调整凛夜的设定」「她的性格要改一下」

更新流程：
1. 读取旧 `manifest.json`，确认当前版本状态
2. 询问用户要调整哪些维度
3. 只重新共创受影响的维度
4. 更新对应的 research 文件和 distillation.md
5. 更新 SKILL.md
6. 更新 manifest.json 的日期
7. 重新质量自检

更新时不得：
- 不经用户确认直接覆盖旧设定
- 删除旧的创作记录（应追加修订记录）

## 6. 质量验证

Phase 6 自动执行以下检查：
- 行为模式数量 ≥ 3
- 表达质感包含至少 3 种情境的示例对白
- 关键场景 ≥ 5 个
- SKILL.md 包含所有 9 个章节
- manifest.json 必填字段完整
- 每个阶段有对应的 research 文件
- sources.json 记录完整

验证结果写入 `references/quality-report.json`。

## 7. 技术实现

### 7.1 形态

OCSP 是一个 Skill，和 `csp` 同级，通过 `/ocsp` 触发。在对话中运行，分阶段交互。

### 7.2 依赖

- 无需外部 Python 脚本（CSP 的脚本用于互联网检索，OCSP 不需要）
- 纯对话驱动，由 Skill 指令控制流程
- OCSP 自带 `skill-template.md` 作为 SKILL.md 模板参考（格式兼容 CSP 模板，但包含创作声明等 OCSP 专有章节）

### 7.3 文件操作

- 生成角色时，先在临时工作区创建 research 文件
- Phase 6 一次性组装最终目录结构
- 生成的角色 Skill 最终存入 `.claude/skills/<character-slug>/`（和 CSP 产出同一位置，可直接部署使用）

### 7.4 错误处理

| 情况 | 处理 |
|------|------|
| 描述过于模糊 | Phase 0 回显时标注「信息不足」，请用户补充 |
| 描述包含矛盾 | 回显矛盾点，请用户澄清 |
| 用户在 Phase 1-5 中大量推翻首稿 | 正常流程，AI 重新生成该维度后再次确认 |
| 用户中途放弃 | 已锁定的阶段保留，可后续恢复 |

## 8. 品味守则（继承自 CSP）

| 原则 | 一句话 |
|------|--------|
| 行为 > 形容词 | 描述「做什么」而非「是什么」 |
| 证据 > 印象 | 重要结论可追溯到用户描述或共创确认 |
| 矛盾 > 一致 | 保留角色的内在矛盾 |
| 语境 > 台词 | 每句示例对白说明情境 |
| 口语 > 文章 | Skill 输出的是角色说话方式，不是论文 |
| 人味 > 完美 | 角色可以不完美、不确定、前后矛盾 |

绝不做：
- 用萌属性标签取代行为描述
- 把角色写成完美人设
- 用户说「整体没问题」后还反复追问细节
- 跳过用户确认直接锁定设定
